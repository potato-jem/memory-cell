Memory Cell — Layer 1 Implementation Plan

What We're Building
A single playable situation proving the core signal routing mechanic is fun. One bacterial infection, one body, complete loop from first signal to resolution or collapse. Target session: 10-15 minutes.

Tech Stack & Setup
React + Vite + Tailwind CSS. Standard scaffolding via npm create vite. No backend, no persistence, all state in React. Single deployable static site.

File Responsibilities
Data Layer — pure data, no logic
data/nodes.js
Defines the 10 body nodes and their connections. Each node has: id, label, position (SVG coordinates for body map), connections to other nodes, signal speed (affects delay on arriving signals), damage weight (how much collateral damage here costs coherence), and special flags (isHQ for spleen, isBottleneck for lymph clusters, isCellSource for bone marrow, isSystemic for blood).
The 10 nodes: Throat, Lungs, Gut, Liver, Spleen, Left Lymph Cluster, Right Lymph Cluster, Bone Marrow, Blood, Peripheral.
Connections are not fully connected — topology matters strategically. Gut connects to Liver, Spleen, Blood. Throat connects to Lungs and both Lymph Clusters. Spleen is the most connected node and acts as HQ.
data/signals.js
Library of signal flavour text templates, organised by signal type and confidence level. Types: patrol clear, anomaly detected, threat confirmed, threat expanding, collateral damage, false alarm (post-mortem only), resolution. Each type has low/medium/high confidence variants. Text uses a {node} placeholder substituted at generation time. Hand-authored for Layer 1.
data/situations/uninvitedGuest.js
The single authored situation for Layer 1. Defines: starting pathogen location (gut), starting strength, spread rate, spread threshold, signal accuracy rate, false alarm rate, turn limit (40), and a sequence of seeded events that give the situation narrative shape regardless of player action. Seeded events include: first low-confidence gut signal on turn 2, false alarm peripheral signal on turn 4 (the red herring), escalating gut signal on turn 6, spread to liver on turn 12 if unaddressed.

Engine Layer — pure functions, no React, no UI
engine/groundTruth.js
Initialises and advances the hidden simulation. Takes the situation definition and produces the starting ground truth state. Each turn, advances the simulation: grows pathogen strength at active locations, applies responder clearance, triggers spread events when threshold is reached, applies and decays inflammation from responder activity, checks win condition. Returns new ground truth state. Never reads from or writes to React state directly.
engine/pathogen.js
Pathogen-specific behaviour isolated here. Handles: strength growth per turn at unaddressed nodes, spread logic (which adjacent nodes are candidates, seeded vs random spread), clearance rate when responders are present. In Layer 1 only bacterial behaviour is needed — this file is designed to accept other pathogen types in Layer 2.
engine/signalGenerator.js
Translates ground truth into signals each turn. Takes ground truth, deployed cells, situation parameters, and current turn number. Produces the array of new signals for this turn. Logic: check seeded events for this turn, generate organic signals from patrolled and threatened nodes, apply signal accuracy rate (some real threats generate no signal — silence is information), apply false alarm rate to patrolled clean nodes, generate collateral damage signals from inflammation above threshold. Dendritic cells returning from a node produce high-confidence signals regardless of accuracy rate.
engine/coherence.js
Computes coherence score from the gap between ground truth and perceived state. Two gap types contribute: undetected threats (ground truth shows threat, perceived state shows nothing — weighted by threat severity) and over-response (perceived state is responding to a location where ground truth shows no threat — weighted by response intensity). Coherence is 100 minus total gap score, floored at 0. Spleen stress adds a multiplier. Returns a number 0-100 and a breakdown of what's driving the gap (used by post-mortem).
engine/cells.js
Manages cell deployment mechanics. Handles: deployment validation (is there capacity? enough tokens?), transit timing (dendritic cells take 2 turns to return), responder effectiveness (full effectiveness with prior dendritic confirmation, reduced without), patrol coverage tracking, recall logic. Returns updated deployment state.

State Layer — game state structure and mutations
state/gameState.js
Defines the complete game state shape. Two parallel structures that must never be conflated: groundTruth (the hidden simulation) and perceivedState (the player's working model). Also holds: deployedCells, attention tokens (current and per-turn rate), turn counter, coherence score, active signals queue, signal history log, situation reference, win/lose status, and postMortem (null during run, populated on resolution).
state/perceivedState.js
Manages the player's working model. Updated when signals arrive and when routing decisions are made. Each node in perceived state tracks: signals received, current player assessment, response level, scout confirmation status. This is what the right panel and body map display — the player's picture, not reality.
state/actions.js
All state mutations as pure functions. One function per player action: routeSignal(signal, decision), deployDendriticCell(nodeId), deployNeutrophilPatrol(nodeId), deployResponder(nodeId), recallUnit(unitId), endTurn(). Each action: validates the action is legal, deducts attention tokens, returns updated game state. endTurn() triggers the engine layer — advances ground truth, generates new signals, recomputes coherence, checks win/lose conditions.

Component Layer — UI only, no game logic
GameShell.jsx
Top-level layout. Three-panel structure: body map left, signal console centre, working model right. Holds the single useReducer for game state. Passes state and dispatch down to panels. Also renders the PostMortem overlay when game is resolved. Header bar showing turn number, attention tokens remaining, coherence readout.
BodyMap.jsx
SVG rendering of the body. Human silhouette outline (simple, not detailed). 10 nodes rendered as circles at their defined positions. Connecting lines between nodes showing lymphatic topology. Node visual states: clear (grey), signalled (yellow), patrolled (blue outline), scouted (blue filled — dendritic returned with intel), responding (red), dark (no macrophage coverage, dimmed). Clicking a node selects it and shows its history in the working model panel. No ground truth visible during play.
SignalConsole.jsx
The primary interaction surface. Active signal queue — signals awaiting routing decision. Each signal card shows: source node name, source cell type, confidence level (low/medium/high), delay indicator, flavour text. Four routing buttons per signal: Forward (cost 1), Amplify (cost 2), Suppress (cost 1), Quarantine (cost 1). Attention token counter at top — prominent. End Turn button. Aged/unrouted signals visually distinguished. Signal history below active queue, scrollable, showing past signals and their routing decisions.
WorkingModel.jsx
The player's current picture of the body. Node list showing perceived state of each: clean, under investigation, threat suspected, threat confirmed, responding, resolved. Filters for showing only nodes with activity. Coherence readout with simple visual bar. Deployed cell summary — where each unit currently is and what it's doing.
PostMortem.jsx
Shown on win or lose. Overlays the main UI. Shows: outcome message (one of four failure modes or win), coherence graph across the run as a simple line chart, body map replay showing ground truth state at final turn with comparison to perceived state, signal log annotated retrospectively (each signal marked accurate / inaccurate / false alarm), 2-3 key decision points called out by turn number with plain language explanation of consequence. Restart button.

Key Design Constraints For Implementation
Ground truth and perceived state are always separate data structures. The coherence calculation is the only place they meet. No shortcuts that merge them even temporarily.
Routing is the intervention. The engine must model this correctly — forwarding a signal triggers a response that generates new ground truth changes and new signals. The loop is: player routes → engine advances → new signals arrive shaped by what player just did.
Silence is information. The signal generator deliberately produces no signal from real threats sometimes (based on signal accuracy rate). The absence of a signal from a patrolled node is meaningful. The UI should make this legible — "no signal received from gut this turn" rather than just nothing.
Four failure modes must all be reachable. The situation and engine must be tested to confirm all four endings are achievable through distinct playstyle errors: missing the threat, being overwhelmed, over-responding, and running out of turns.

Attention Token Economy
Base rate: 5 tokens per turn.
Costs: Forward signal (1), Amplify signal (2), Suppress signal (1), Quarantine signal (1), Deploy dendritic cell (2), Deploy neutrophil patrol (1), Deploy responder (3), Recall unit (0).
Unrouted signals carry over to next turn but age — delay increases, confidence may degrade. Player can intentionally defer cheap signals to spend tokens on more important actions.
This economy needs playtesting immediately — 5 tokens is a starting point, not a final number.

What The Post-Mortem Must Show
This is design-critical even in Layer 1. The post-mortem is where learning happens and where the gap between signal and reality becomes visible. It must show:

What the ground truth actually was at every turn, compared to player's perceived state
Which signals were accurate, which were inaccurate, which were false alarms
The 2-3 most consequential decision points with plain language annotation
The coherence trajectory across the run
The specific failure mode that ended the run, or the win condition that was met


Build Order
Build in this sequence — each step should be testable before proceeding:

Data files (nodes, signals, situation) — no logic, just verify structure is correct
Engine functions with unit tests — verify ground truth advances correctly, signals generate correctly, coherence calculates correctly, all independently of UI
State layer — wire engine functions into state mutations, verify actions produce correct state transitions
GameShell with placeholder panels — verify state flows correctly top to bottom
SignalConsole — the most important panel, build and test routing interactions first
BodyMap — SVG rendering of node states
WorkingModel — perceived state display
PostMortem — build last, depends on complete run data
Full playtest loop — play until all four failure modes are confirmed reachable


Definition Of Done For Layer 1

Complete run playable from first signal to resolution or collapse
All four failure modes reachable and producing distinct end screens
Post-mortem produces a genuine "oh, that's what was happening" moment
Attention token economy feels meaningful — never trivially flush, never impossibly tight
Session length 10-15 minutes
No known bugs in ground truth / perceived state separation
At least 3 playtests with the specific question: does routing feel like a real strategic decision?
