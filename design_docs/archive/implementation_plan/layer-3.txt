Memory Cell — Layer 3: Scar & Upgrade System
============================================================

Goal: make individual situation outcomes feel consequential. Situations now leave
marks — positive and negative — that affect subsequent situations in the same session.

Playtest question: Do scars feel interesting rather than just punishing? Do upgrades
feel earned rather than arbitrary?

Done when: A player can describe their current body's "character" — its specific
strengths and vulnerabilities — and feel that character is the result of their decisions.

MECHANICAL DESIGN
-----------------

Scars
  Scars are persistent negative state generated by how a situation resolves.
  They fall into two categories:

  Temporary scars — last N situations (usually 1–3), then fade.
  Permanent scars  — require active repair via a matching situation type.

  Scar effects are mechanical, not just narrative:
    inflamed_node      Specific node generates +1–2 false-positive signals/turn.
                       Repair: deploy 2+ macrophages to that node and hold for 2 turns.

    scar_tissue        Specific node: cell clearance rate –30%.
                       Permanent. Repair: resolve a bacterial threat at that exact node.

    hyperactive        Autoimmune tendency. Any SUSPECTED node costs +1 token to deploy to.
                       Spleen stress threshold –10.
                       Repair: resolve a friendlyFire situation using only Hold/Dismiss
                       (no offensive cells deployed).

    exhausted_cells    Max simultaneous deployed cells –1 (floor: 3).
                       Temporary: lasts 2 situations.

    signal_fatigue     Signals from one specific node arrive 1 turn delayed.
                       Temporary: lasts 1 situation.

    depleted_lymph     Attention tokens per turn: 4 instead of 5.
                       Temporary: lasts 2 situations. Repair: pass an entire situation
                       turn without deploying any cells.

    chronic_inflammation  Health bleeds –1/turn for every 3 cells currently deployed
                          (rounds down). Permanent if inflamed_node went unrepaired
                          for 2+ situations.


Upgrades
  Upgrades are persistent positive state earned by clean resolution or specific
  tactical decisions.

    memory_[threatType]  Signals from that threat type arrive with confidence +1 tier
                         (low→medium, medium→high). Does not stack with Memory Bank,
                         but compounds it.

    primed_[nodeId]      That node's signals arrive 1 turn earlier.
                         Earned by: sending a dendritic to that node and getting a
                         clean report (no threat found).

    extra_tokens         Attention tokens per turn: 6.
                         Earned by: completing a situation with ≥4 tokens unspent
                         at the final turn.

    fast_transit         All cell transit times –1 turn (floor: 1).
                         Earned by: deploying 3+ different cell types in a single turn.

    amplified_scouts     Dendritic cells return 1 turn earlier.
                         Earned by: a dendritic correctly classifying a mimic.

    resilient_[nodeId]   That node's damage weight –20%.
                         Earned by: a node surviving a CONFIRMED threat without health
                         dropping below 60.

    trained_neutrophils  Neutrophil clearance rate +20%.
                         Earned by: neutrophils clearing a bacterial threat before
                         macrophages needed to be deployed.


Scar/Upgrade Interaction Pairs (Identity Expressions)
  When a scar and a complementary upgrade are both active simultaneously,
  they combine into a named "body identity" with a distinct mechanical character.
  These are NOT automatically granted — the player accumulates them through play.

  hyperactive + memory_[type]
    → "Sensitized [type]" identity
    Effect: That specific threat type clears 1 turn faster. Token cost on SUSPECTED
    nodes drops back to normal (scar partially neutralised). But autoimmune risk
    remains +10% baseline.

  inflamed_node + primed_node (same node)
    → "Vigilant" identity for that node
    Effect: That node generates 2× signals per turn — signal count doubles, but
    half are annotated "residual" (player can learn to filter). The node also
    self-reports WATCHING status automatically on turn 1 of any situation.

  depleted_lymph + extra_tokens
    → "Efficient" identity
    Effect: Net tokens = 5/turn (scar and upgrade cancel). But additionally, any
    turn where you spend exactly 3 tokens grants +1 free token next turn (efficiency
    reward for precise spending).

  exhausted_cells + fast_transit
    → "Lean & Fast" identity
    Effect: Cell cap reduced by 1, but all cells move at minimum 1 turn transit.
    Encourages aggressive rotation rather than large static deployments.

IMPLEMENTATION PLAN
--------------------

Step 1 — Data: scar and upgrade definitions
  New file: src/data/scars.js
    Export SCAR_TYPES object with entries:
      id, label, description, category ('temporary'|'permanent'),
      defaultDuration (turns for temporary), effect (object describing mechanical
      modification — see below), repairCondition (string key used by scarSystem),
      nodeScoped (bool — true for inflamed_node, scar_tissue, etc.)

    Export UPGRADE_TYPES object similarly:
      id, label, description, earnCondition (string key), nodeScoped (bool)

    Export IDENTITY_PAIRS array:
      { scarId, upgradeId, name, description, modifiers }

  Effect object schema (used by engine to read modifications):
    {
      tokenDelta: -1,           // tokens/turn adjustment
      cellCapDelta: -1,         // max deployed cells adjustment
      transitDelta: -1,         // turns in transit adjustment
      signalDelayNodeId: null,  // nodeId that gets delayed signals
      falsePosNodeId: null,     // nodeId that generates extra false positives
      healthBleedPerCells: 3,   // health bleed: 1hp per N cells deployed
      clearanceMult: 0.7,       // multiplier on clearance rates (per node if nodeScoped)
      autoimmuneCostDelta: 1,   // extra token cost when deploying to SUSPECTED node
      spleenStressDelta: -10,   // adjustment to spleen stress threshold
      dendriticReturnDelta: -1, // turns faster for dendritic return
      neutrophilClearanceMult: 1.2, // neutrophil clearance multiplier
    }
    (All fields optional; absent = no effect)


Step 2 — Engine: scar system logic
  New file: src/engine/scarSystem.js
    computeActiveScarEffects(activeScars, activeUpgrades)
      → Merges all active scar and upgrade effects into a single "modifiers" object
        that other engine functions will read. Also checks for identity pairs.
      → Returns: { tokenDelta, cellCapDelta, transitDelta, healthBleedRate,
                   falsePosNodes, delayedNodes, clearanceMults, autoimmuneCostDelta,
                   spleenStressDelta, identities[] }

    generateSituationOutcome(situationState, groundTruth, perceivedState, deployedCells)
      → Called when a situation resolves (win or loss). Evaluates what scars/upgrades
        were earned based on how the situation played out.
      → Checks: final health score, turns taken, cell types used, whether dendritic
        was sent, whether false positives were chased, whether threats were missed.
      → Returns: { scarsEarned: ScarInstance[], upgradesEarned: UpgradeInstance[] }

    checkRepairConditions(activeScars, situationResult, actions)
      → Checks whether any active permanent scars have their repair condition met
        by the just-completed situation.
      → Returns: scarIdsRepaired[]

    ScarInstance: { id, nodeId (optional), turnsRemaining (null if permanent),
                    earnedOnSituation, label }
    UpgradeInstance: { id, nodeId (optional), turnsRemaining (null if permanent),
                       earnedOnSituation, label }


Step 3 — State: gameState and actions updates
  Modify src/state/gameState.js
    Add to initial state:
      activeScars: []        // ScarInstance[]
      activeUpgrades: []     // UpgradeInstance[]
      activeMods: {}         // computed by computeActiveScarEffects each turn
      situationHistory: []   // { situationId, outcome, scarsEarned, upgradesEarned }

  Modify src/state/actions.js
    Add ACTION_TYPES:
      APPLY_SCAR          // used by outcome system and test tools
      APPLY_UPGRADE
      REPAIR_SCAR
      SITUATION_COMPLETE  // triggers outcome generation, scar/upgrade award

    In handleEndTurn:
      1. Read activeMods from state.activeMods.
      2. Pass activeMods to advanceCells() for transit/clearance adjustments.
      3. Pass activeMods to generateSignals() for delay/false-pos adjustments.
      4. Apply health bleed: if activeMods.healthBleedRate and deployed cell
         count >= threshold, subtract hp.
      5. Adjust token regen: state.attentionTokens = Math.min(
           tokenCap, tokens + 5 + (activeMods.tokenDelta ?? 0))
      6. Check if any situation is now complete; if so dispatch SITUATION_COMPLETE.

    In handleSituationComplete:
      1. Call generateSituationOutcome() → { scarsEarned, upgradesEarned }
      2. Call checkRepairConditions() → scarIdsRepaired
      3. Decrement turnsRemaining on temporary scars; remove expired ones.
      4. Remove repaired scars.
      5. Add new scars and upgrades.
      6. Recompute activeMods via computeActiveScarEffects().
      7. Push to situationHistory.
      8. Set game phase to SITUATION_OUTCOME (triggers outcome screen).

    In handleDeployCell:
      Read activeMods.autoimmuneCostDelta — if node status is SUSPECTED,
      add extra token cost before checking affordability.

    Cell cap enforcement:
      Max cells = 6 + (activeMods.cellCapDelta ?? 0), floor 3.

    In computeCoherence call:
      Pass activeMods.spleenStressDelta to adjust stress threshold.


Step 4 — Engine: wire mods into existing engines
  Modify src/engine/cells.js
    advanceCells(deployedCells, groundTruth, turn, mods)
      Use mods.transitDelta to adjust inTransit threshold.
      Use mods.clearanceMults[nodeId] (or default 1) in getClearancePower().
      Use mods.neutrophilClearanceMult if cell type is neutrophil.

  Modify src/engine/signalGenerator.js
    generateSignals(..., mods)
      If mods.delayedNodes includes current nodeId, push signal to next turn's
      queue rather than current (add a pendingSignals buffer to groundTruth or
      pass through state).
      If mods.falsePosNodes includes current nodeId, add 1 extra patrol_clear
      or anomaly_detected signal per turn (50/50) with low confidence.

  Modify src/engine/coherence.js
    computeCoherence(groundTruth, perceivedState, mods)
      If mods.spleenStressDelta, offset the spleen_stress gap threshold by that amount.

Step 6 — Triage pressure: concurrent situation wiring
  Modify src/state/gameState.js and actions.js to properly support 2–3 concurrent
  situations active simultaneously.

  Currently situationStates is an array but mostly used as single element.
  Make the concurrent path real:
    - Multiple situations each generate signals per turn.
    - Signals from all situations pooled into activeSignals, distinguished by
      situationId field (already present).
    - Attention tokens are shared across all concurrent situations (triage = you
      can't respond to everything fully).
    - End turn advances ALL active situations' groundTruth.
    - Health score = mean coherence across all active situations (already designed
      this way in coherence.js).
    - Each situation completes independently; outcome generated separately.

  Modify src/components/SituationSelector.jsx
    Enable the "concurrent" mode toggle for situation pairs already defined.
    When concurrent: starts two situations simultaneously.
    Instruction text: "You have two situations developing. One attention pool."

BALANCE NOTES
--------------

Scar accumulation rate:
  A clean win (threat cleared, health >70, no excess cells): no scar, 1 upgrade.
  A messy win (health 50–70, excess cells deployed): 1 temp scar, maybe 1 upgrade.
  An overreaction (health >70 but massive deployment): hyperactive temp scar.
  A near-miss (health 30–50): 1 temp scar + 1 perm scar.
  A loss (health <30 or threat unchecked): 1–2 perm scars.

Scar density target:
  After 3 situations, a player should have 1–2 active scars and 1–2 upgrades.
  After 6, they should have 2–4 scars and 2–3 upgrades — a distinct character profile.
  At no point should scars make the game unplayable; they add flavour and constraint,
  not brick walls.

Token economy with scars:
  Even with depleted_lymph (4 tokens) the game should feel tactical, not impossible.
  Design situations with token budgets in mind: a standard turn needs ~3 tokens,
  so 4 still gives slack. Only with exhausted_cells AND depleted_lymph simultaneously
  does it feel genuinely pressured (and that combination should be rare without the
  Lean & Fast identity softening it).
